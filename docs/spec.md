# シールカウンター 技術仕様書

> ヤマザキ春のパンまつり 2026 - シールポイント自動認識 Web アプリ
>
> 最終更新: 2026-02-26
> レビュー: Gemini (設計担当) — Approved with suggestions

## 1. プロジェクト概要

台紙等に貼られたヤマザキ春のパンまつりシールをスマートフォンのカメラで認識し、ポイントを自動集計するモバイル Web アプリケーション。

### 目標

- カメラ映像からシールを検出し、額面（点数）を自動判定
- 有効年度（2026年）のシールのみカウント、旧年度シールは除外表示
- サーバー不要のクライアントサイド完結型

```
│                   │      Web Worker           │   │
│                   │  ONNX Runtime Web         │   │
│                   │  YOLOv8-nano 推論         │   │

│                   └──────────────────────────┘   │
│                                                  │
│  ┌──────────────────────────────────────────┐    │
│  │  IndexedDB / LocalStorage                │    │
│  │  モデルキャッシュ（バージョン管理）       │    │
│  │  検出履歴 + ポイント状態                  │    │
│  └──────────────────────────────────────────┘    │
└──────────────────────────────────────────────────┘
```

### 技術スタック

| レイヤー | 技術 | 理由 |
|----------|------|------|
| UI | HTML + TailwindCSS (CDN) + Vanilla JS | 既存MVPと統一、ビルド不要 |
| カメラ | getUserMedia API | 標準Web API、iOS/Android対応 |
| 推論エンジン | ONNX Runtime Web | WebGL/WASM 対応、YOLOv8 との親和性高 |
| モデル | YOLOv8-nano (ONNX) | 軽量（~6MB）、検出+分類を1ステージ |
| 推論実行 | Web Worker | メインスレッド非ブロック |
| 永続化 | LocalStorage (状態) + IndexedDB (モデルキャッシュ) | オフライン対応 |
| フィードバック | Vibration API + Audio API | 検出成功時の触覚・聴覚フィードバック |

### カメラ解像度設定

```javascript
// getUserMedia constraints
const constraints = {
  video: {
    facingMode: 'environment',
    width: { ideal: 1280 },
    height: { ideal: 720 }
  }
};
```

> **設計根拠** (Gemini レビュー): 解像度が高すぎるとパフォーマンスを圧迫し、低すぎると検出精度が低下する。1280x720 は性能と精度のバランスが最適。

## 4. 検出パイプライン

### Phase 1: 半自動スキャン

```
ユーザー操作          システム処理
─────────────────────────────────────────────
カメラ起動        →  getUserMedia でストリーム取得
                     video 要素にプレビュー表示
                     撮影ガイドオーバーレイ表示
                     ↓
「スキャン」押下  →  ローディングスピナー表示
                     Canvas に現フレーム描画
                     ImageData → Float32Array 変換
                     Web Worker に転送
                     ↓
                     ONNX Runtime で推論実行
                     NMS (IoU 0.45, conf 0.25)
                     ↓
                     検出結果を返却
                     ↓
結果確認          ←  ローディング解除
                     バウンディングボックス描画
                     各シールに額面+年度ラベル表示
                     検出成功時: 効果音 + 振動フィードバック
                     ↓
「確定」押下      →  有効シール(2026年)のポイントを加算
                     LocalStorage に保存
```

### Phase 2: 連続映像処理

```
カメラストリーム → フレーム取得 (requestAnimationFrame)
                    ↓
              スロットリング (200ms 間隔 = 5fps)
                    ↓
              Web Worker で推論
                    ↓
              IoU ベース トラッキング
              (前フレームの検出結果と照合)
                    ↓
              新規検出 → UI に追加 + フィードバック
              既存検出 → 位置更新
              消失検出 → N フレーム後に除外
                    ↓
              オーバーレイ描画 (メインスレッド)
```

### 前処理

```javascript
// 入力画像 → モデル入力テンソル変換
1. Canvas に描画 (リサイズ: 640x640, letterbox padding)
   ※ padding 色: RGB(114, 114, 114) — 学習時と推論時で必ず一致させる
2. getImageData() → RGBA pixel array
3. RGB 抽出 + 正規化 (0-255 → 0.0-1.0)
4. CHW 形式に転置 [1, 3, 640, 640]
5. Float32Array として Web Worker に transferable 転送
```

> **注意** (Gemini レビュー): letterbox padding の色は学習時の Ultralytics デフォルト `(114, 114, 114)` と一致させること。不一致は精度低下の原因となる。

### 後処理 (NMS)

```
推論出力 [1, 8400, 11] (4 bbox + 7 classes)
    ↓
各検出ボックス: [x_center, y_center, w, h, class_scores...]
    ↓
信頼度フィルタ (threshold: 0.25)
    ↓
Non-Maximum Suppression (IoU threshold: 0.45)
    ↓
座標を元画像サイズにスケール変換
    ↓
検出結果: [{bbox, class, confidence}, ...]
```

## 5. クラス定義

### 検出クラス (7クラス)

| class_id | ラベル | 説明 | UI表示色 |
|----------|--------|------|----------|
| 0 | `seal_05` | 2026年 0.5点シール | 緑 `#22c55e` |
| 1 | `seal_1` | 2026年 1点シール | 緑 |
| 2 | `seal_15` | 2026年 1.5点シール | 緑 |
| 3 | `seal_2` | 2026年 2点シール | 緑 |
| 4 | `seal_25` | 2026年 2.5点シール | 緑 |
| 5 | `seal_3` | 2026年 3点シール | 緑 |
| 6 | `seal_old` | 旧年度シール（2025以前） | オレンジ `#f97316` |

> **Pasco シールについて**: 現段階では学習対象外（無視）。将来 Phase 3 で Pasco キャンペーン用の検出クラスを追加し、別途カウント機能を実装予定。モデル設計時は Pasco クラス追加に備え、クラス数の拡張が容易な構成とする。

> **seal_old の代替戦略** (Gemini レビュー): `seal_old` の精度が上がりにくい場合は、`seal_2025` / `seal_2024` のように年度別に分けて学習し、後処理で「2026年以外」としてグループ化する方法も選択肢として残す。まずは `seal_old` で進め、評価結果を見て判断する。

### 年度識別の戦略

**アプローチ**: 年度をクラスラベルに組み込む（2026年の6額面 vs 旧年度）

**根拠**:
- 年度ごとにデザインが微妙に異なる（白枠有無、色調、フォント等）
- 分類モデルがデザイン特徴を学習することで年度を暗黙的に識別
- 2025/2024年のシール画像をメルカリ等から収集し `seal_old` として学習

**UI表現**:
- 2026年判定 → 緑枠 + ポイント値表示 → 合計に加算
- 旧年度判定 → オレンジ枠 + 「旧年度？」ラベル → 加算せず注意喚起
- 低信頼度 (< 0.5) → 黄枠 + 「要確認」→ ユーザータップで手動判定

### 概要フロー

```
[データ収集] → [アノテーション] → [データ拡張] → [学習] → [評価] → [エクスポート]
     ↑              ↑                                          │
```

### 6.1 データ収集

#### ソース1: 公式画像（グラウンドトゥルース）

```
URL パターン:
  https://www.yamazakipan.co.jp/campaign/spring/assets/img/seal/{value}.png
  value = 05, 1, 15, 2, 25, 3

台紙 PDF:
  https://www.yamazakipan.co.jp/campaign/spring/assets/pdf/daishi2026.pdf
```

- 6枚の正規シール画像 → 合成データ生成のベース素材
- 台紙レイアウト → 背景合成用テンプレート

#### ソース2: マーケットプレイス（実世界データ）

```
検索キーワード:
  メルカリ:  "春のパンまつり シール 台紙"
  ヤフオク:  "ヤマザキ パンまつり シール"

対象年度:
  2026 → seal_05〜seal_3 クラス
  2025, 2024 → seal_old クラス
```

- 実際の貼り付け状態（照明、角度、テープ反射等）を含む
- シワ・破れ・汚れなど物理的劣化状態の画像を積極的に収集
- 目標: 2026年 100枚以上、旧年度 50枚以上

#### ソース3: ブログ・SNS（補助データ）

```
検索キーワード:
  "パンまつり シール 集め"
  "パンまつり 台紙 写真"
  "春のパンまつり 2026"
```

- 多様な撮影環境のバリエーション確保

#### ネガティブサンプリング（誤検出抑制）

> **Gemini レビュー指摘**: 背景からの誤検出（False Positives）を減らすため、以下の画像を**ラベル付けせずに**学習データに含める。

```
ネガティブサンプル（ラベルなし画像）:
  - Pasco シールが写っている画像（将来の学習に備え raw/ にも保存）
  - 円形の無関係な物体（コースター、シール状の装飾等）
  - 台紙のみ（シール未貼付）の画像
  - 日常的な背景画像（テーブル、冷蔵庫等）

目標: ネガティブサンプル 50枚以上
```

### 6.2 合成データ生成

公式シール画像 + 多様な背景で合成データを自動生成:

```python
# 合成パラメータ
backgrounds = ["台紙", "冷蔵庫", "テーブル", "手のひら", "ランダムテクスチャ"]
seals_per_image = random.randint(1, 12)  # 1〜12枚のシール配置
transforms:
  - rotation: -30° 〜 +30°
  - scale: 0.5x 〜 2.0x
  - brightness: 0.6 〜 1.4
  - contrast: 0.7 〜 1.3
  - gaussian_blur: 0 〜 2px
  - gaussian_noise: sigma 0 〜 10
  - perspective_warp: 軽微
  - jpeg_compression: quality 50-95
  - elastic_deformation: 軽微（シワ・折れ曲がり再現）
  - stain_overlay: ランダム汚れテクスチャ（任意）
```

> **Gemini レビュー指摘**: シワ、破れ、汚れ対策として elastic deformations（弾性変形）と汚れテクスチャオーバーレイを合成パラメータに追加。

**目標**: 合成データ 500枚以上（シールインスタンス 3,000以上）

**クラスバランス管理**: 各額面クラスのインスタンス数を均等に保つ。特に 0.5点や3点など実世界データが少ないクラスは合成データで重点的に補強する。

### 6.3 アノテーション

**ツール**: Roboflow (推奨) または CVAT

```yaml
# Roboflow プロジェクト設定
project_name: yamazaki-seal-detector
annotation_format: YOLOv8
classes:
  - seal_05
  - seal_1
  - seal_15
  - seal_2
  - seal_25
  - seal_3
  - seal_old

# ラベリングガイドライン
rules:
  - シール全体をタイトにバウンディングボックスで囲む
  - 一部が隠れているシールも可視部分50%以上なら含める
  - テープが被っているシールもシールとしてアノテーション
  - Pasco シールは現段階ではアノテーション対象外（スキップ）
  - 不鮮明で判読不能なシールはスキップ
  - アノテーション後、クラスごとのインスタンス数を確認し偏りを是正
```

### 6.4 学習

```yaml
# config.yaml
model: yolov8n.pt          # YOLOv8-nano ベースモデル
data:
  train: ./data/train/images
  val: ./data/val/images
  nc: 7                     # クラス数
  names: [seal_05, seal_1, seal_15, seal_2, seal_25, seal_3, seal_old]

# ハイパーパラメータ
epochs: 100
imgsz: 640
batch: 16
optimizer: AdamW
lr0: 0.001
lrf: 0.01
momentum: 0.937
weight_decay: 0.0005
warmup_epochs: 3
augment: true               # Ultralytics 組み込み拡張も併用

# データ分割
train: 80%
val: 20%
```

**実行環境**: ローカル GPU (CUDA)

> 時間がかかっても構わないため、ローカル GPU を主軸とする。Colab は使用しない。

### 6.5 評価基準

| メトリクス | 目標値 | 備考 |
|------------|--------|------|
| mAP@0.5 | >= 0.85 | 全クラス平均 |
| mAP@0.5:0.95 | >= 0.60 | 厳密評価 |
| 2026年シール Precision | >= 0.90 | 偽陽性（旧年度を誤カウント）を抑制 |
| 2026年シール Recall | >= 0.85 | 見落としは手動補完で対応可 |
| seal_old Recall | >= 0.70 | ベストエフォート |
| 推論速度 (640x640) | < 100ms | ONNX Runtime Web (WebGL) |
| クラスバランス | 最大/最小比 <= 3:1 | インスタンス数の均衡 |

### 6.6 エクスポート

```bash
# PyTorch → ONNX
yolo export model=best.pt format=onnx imgsz=640 simplify=True opset=13

# ONNX 量子化（オプション: サイズ削減）
python -m onnxruntime.quantization.quantize \
  --input best.onnx \
  --output best_quantized.onnx \
  --quant_format QDQ \
  --per_channel
```

**出力**: `seal-detector.onnx` (~3-6MB)

## 7. UI/UX 設計

### 画面遷移

```
[ホーム画面]
  ├─ ポイント進捗表示（既存）
  ├─ カメラプレビュー領域
  │   ├─ 撮影ガイドオーバーレイ（「シールが全体的に写るように真上から撮影」）
  │   ├─ 「スキャンする」ボタン → [検出結果画面]
  │   └─ （Phase 2）リアルタイムオーバーレイ
  ├─ クイック入力ボタン（既存: 手動追加）
  └─ 登録シール一覧（既存）

[検出結果画面]（モーダルまたは同画面内）
  ├─ 撮影画像 + バウンディングボックスオーバーレイ
  ├─ 検出シール一覧
  │   ├─ 各シール: 額面 + 信頼度 + 年度判定
  │   ├─ タップで除外/復活トグル
  │   └─ 旧年度シールはデフォルト除外（オレンジ表示）
  ├─ 「追加スキャン」→ 別角度から追加撮影
  ├─ 「手動追加」→ 見落としを手動で追加
  └─ 「確定」→ 有効シールをポイントに加算 + フィードバック
```

### バウンディングボックス描画仕様

```
2026年シール (confidence >= 0.5):
  枠: 2px solid #22c55e（緑）
  ラベル: "{額面}点" 背景 #22c55e

旧年度シール:
  枠: 2px dashed #f97316（オレンジ・破線）
  ラベル: "旧年度?" 背景 #f97316

低信頼度 (0.25 <= confidence < 0.5):
  枠: 2px dotted #eab308（黄・点線）
  ラベル: "要確認" 背景 #eab308
```

### 撮影ガイド

> **Gemini レビュー指摘**: ピンボケ・手ブレ・極端な角度は精度低下の主因。撮影ガイドでユーザーの撮影品質を向上させる。

```
カメラプレビュー上に半透明オーバーレイ:
  中央: 四隅ガイド枠（台紙をこの枠に合わせて撮影）
  下部: 「シール全体がはっきり写るように、真上から撮影してください」
  初回のみ表示 → 「今後表示しない」チェックボックス
```

### 検出フィードバック

```
検出成功時:
  - 短い効果音（ビープ音、100ms）
  - Vibration API: navigator.vibrate(50)
  - バウンディングボックスのフェードインアニメーション

スキャン中:
  - ローディングスピナー + 「認識中...」テキスト
  - スキャンボタン無効化（二重実行防止）
```

### モデル読み込みUX

```
初回アクセス:
  「モデルを読み込み中...」プログレスバー表示
  IndexedDB にキャッシュ保存（バージョン付き）

2回目以降:
  キャッシュから即座に読み込み（< 1秒）
  アプリ内 MODEL_VERSION 定数と IndexedDB のバージョンを比較
  → 不一致なら新モデルを再ダウンロード・更新
```

### モデルバージョン管理

> **Gemini レビュー指摘**: モデル更新時のスムーズな配信のため、明示的なバージョン管理を導入。

```javascript
// app.js 内
const MODEL_VERSION = '1.0.0';
const MODEL_URL = './models/seal-detector.onnx';

// IndexedDB キャッシュスキーマ
{
  key: 'seal-detector',
  version: '1.0.0',      // MODEL_VERSION と比較
  data: ArrayBuffer,       // ONNX モデルバイナリ
  cachedAt: timestamp
}
```

## 10. リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| 2026年シールのデザインが入手困難 | 学習データ不足 | キャンペーン開始後に実物を撮影、合成データで初期モデルを構築し漸進的に改善 |
| スマホの計算能力不足 | 推論が遅い | 量子化 (INT8)、入力解像度縮小 (640→320)、推論間隔調整 |
| 低スペック端末での WASM フォールバック | 1000ms 超の推論遅延 | ローディングスピナー表示、Phase 1 は半自動のため許容範囲 |
| 照明条件のバリエーション | 検出精度低下 | データ拡張で明暗・色温度バリエーション追加、ユーザーに撮影ガイド表示 |
| ユーザーの撮影品質（ピンボケ・手ブレ・角度） | 検出精度低下 | 撮影ガイド UI で真上からの撮影を促す、データ拡張でぼかし耐性を強化 |
| シールの物理的劣化（シワ・破れ・汚れ） | 検出困難 | elastic deformation + 汚れテクスチャでデータ拡張、実世界の劣化画像を積極収集 |
| ONNX Runtime Web の互換性 | 一部端末で動作しない | TensorFlow.js をフォールバックとして準備 |
| カットアウト型シール（不規則形状） | 検出困難 | 学習データにカットアウト型を意図的に含める |
| Pasco シールの誤検出 | ヤマザキとして誤カウント | ネガティブサンプルとして学習データに含め誤検出を抑制（将来 Phase 3 で正式対応） |
| クラス不均衡 | 少数クラスの精度低下 | 合成データで重点補強、クラスバランス比 3:1 以内を維持 |
| letterbox padding 色の不一致 | 精度低下 | 学習・推論ともに RGB(114,114,114) を使用、仕様に明記 |

## 付録A: 前回リサーチからの知見

- シールはピンク色基調、Pasco競合は青色
- 2025年は2点・2.5点のみ白枠あり（食パン製品マーカー仮説）
- 物理寸法は非公開 → サイズ不変検出が必須
- 既存Android競合アプリは手動入力のみ（自動認識なし）
- 台紙は30枠グリッド構成
- シール窃盗によるパッケージ破損シールが存在
- メルカリに2024/2025年の実物写真が多数出品中
